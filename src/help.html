Project Setup

This project was developed on Mac OS X and tested with the most recent Chrome browser.  To setup on a Mac, clone the project from Github into a subfolder in your Sites folder, for example Sites/Znode.  Then in a browser navigate to http://localhost/~username/, followed by the directory you cloned the project into, followed by the “src” subfolder in the project, for example: http://localhost/~username/Znode/src

Top Menu

The top menu of the project allows you to Save or Open files, Clear All the data in the workspace, or look at the Help document.  To view the example project using the jsPlatformer game, open the “test” file.  Note that the Resources view, which displays previews of images in the project, assumes that the game files are loaded into a subdirectory named “MainMenu” (or else Resources view will not be able to show the images from the project).  To the right of the Filename box are buttons to switch between each of the views available: Source Code view, Inheritance view, Function Calls view, Global Variables view, Composition view, and Resources view.

Source Code View

The original Znode application could not properly save text in nodes with certain special characters.  In my implementation, I still save everything to JSON files, but I use the JSON.stringify() function in order to ensure proper formatting of my data structure.  The allows me to convert all of the project data, including large pieces of source code, into a JSON-formatted string with one command.

 

In Source Code view, the background can be clicked to open a new source code box.  Each box can be moved and resized  In the top bar of each box, you can click “File:” to either name a new class or to choose one of the classes already in the project.  Source code can be copy-and-pasted in to each box, and as long as the box has been named (by selecting “New...” from the file menu), the code will be saved when you save your file.

 

There is a horizontal scroll bar at the bottom of the page.  Click this to move all of the source code boxes side to side in order to reorganize them or to get a clear space to add a new box.  The further your pointer is from the center of the screen when you click, the faster the view will move horizontally.  The boxes also fade out when they get close to the edge of the screen.

Inheritance View

When you select a class in Inheritance view, it will display that class (in orange) along with any subclasses or superclass (in yellow).  Any classes, superclasses, or subclasses can be clicked to display the source code for that class (if it has been pasted into Source view).  The superclass functions and variables used by the class are also shown, and when clicked, they will be highlighted in the source box.  In this and other views, there are input boxes, “+” buttons, and “X” buttons to add or remove entries in the various types of information for the project.  Note that the “+” button at the bottom of the inheritance tree is used to add subclasses.  Figure 2 shows a screenshot of Inheritance view.

Potential Improvements

One major improvement that this visualizer could use is the ability to parse text.  If that were implemented, the function calls, class hierarchies, and much of the other views could be constructed automatically.  The source code view could also use different colors of text for keywords, variables, comments, etc., making it easier to quickly understand the code.

 

Another beneficial feature would be the ability to set status flags or icons for each class to signify what work needs to be done.  Then a to-do list could be constructed and notes could be added.  As changes are made, they could be logged and graphed on a timeline, showing the progress being made as new pieces of the project are added or bugs are fixed.

 

Lastly, it would be useful if the software visualizer allowed easy renaming or moving of functions and variables, so that any changes are reflected in all views, including the source code.
